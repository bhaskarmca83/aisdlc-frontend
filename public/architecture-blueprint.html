E2E SDLC Pipeline — Click any stage to expand
How it flows: Each stage is handled by a specialized LangGraph node (agent). The orchestrator graph routes between them via conditional edges. A shared memory store persists context across all stages — so the Implementation agent knows what the Design agent decided. Click any stage to see its tools, inputs, and outputs.
Agent Registry — 9 Specialized Agents
Agent = LangGraph node with a dedicated system prompt, a bound LLM provider, a tool set, and typed input/output schema. Agents communicate only through the shared SDLCState object — never directly.
Memory Architecture — Story-to-Story Context
Critical design: When Story 2 begins executing, it receives the full accumulated context from Story 1 — coding conventions established, architecture patterns decided, test frameworks chosen, env vars set. This prevents agents from re-inventing the wheel on every story.

Repo Registry — Multi-Repo Story Assignment
Multi-repo awareness: A single story can span multiple repos. The orchestrator reads the repo_registry at runtime and determines which repos the Implementation Agent needs to clone, modify, and PR. Each repo has its own pipeline config, language, and framework context loaded into agent memory.

Story → Repo Routing Logic
How the orchestrator assigns repos to a story
# RepoRouter.resolve(story: Story) → List[RepoConfig]

if story.tags.contains("api") → assign backend_repo
if story.tags.contains("ui") → assign frontend_repo
if story.tags.contains("infra") → assign infra_repo
if story.acceptance_criteria mentions "database" → assign backend_repo
if story.acceptance_criteria mentions "screen" or "page" → assign frontend_repo

# LLM fallback: if rules are ambiguous, ask the Design Agent
else: design_agent.classify_repos(story) → List[repo_id]
LLM Provider Architecture — Bedrock + Vertex AI
Provider-per-agent routing: Each agent is configured with a primary and fallback provider. The ProviderFactory handles auth, retry, rate limiting, and cost tracking. Agents are decoupled from provider details via the ChatModel abstraction layer.

ProviderFactory — Abstraction Layer
agents never import boto3 or google-cloud directly
# providers/provider_factory.py

class ProviderFactory:
  def get_model(agent_role: AgentRole) → BaseChatModel:
    config = AGENT_PROVIDER_MAP[agent_role]
    match config.provider:
      case "bedrock": return ChatBedrockConverse(model=config.model)
      case "vertex": return ChatVertexAI(model=config.model)
      case "ollama": return ChatOllama(model=config.model)

# Automatic failover with exponential backoff
  def with_fallback(primary, fallback) → RunnableWithFallbacks
Observability & Monitoring Architecture
Developers see everything in real-time: Every agent event (start, tool call, LLM call, output, error) is streamed via .astream_events(). The monitoring dashboard (Option B artifact) subscribes to this event stream and renders it live.
📡
LangGraph Event Stream
Uses .astream_events(v2) to emit every node entry/exit, tool call, and LLM token as a typed event. WebSocket bridge to the dashboard.
🔍
LangSmith Tracing
Full run traces with token counts, latency, cost per LLM call. Per-agent cost attribution across Bedrock and Vertex.
📊
OpenTelemetry Metrics
Agent execution time, error rates, retry counts, queue depth — all exported to CloudWatch / Google Cloud Monitoring.
🧾
Audit Log
Immutable append-only log of every decision, approval, deployment, and rollback. Postgres. Queryable by story, agent, or time range.
🚨
Alert Engine
Agent failure, deployment rollback, test failure, approval timeout — all routed to Slack/PagerDuty with full context attached.
🔐
RBAC Gate
Approval gates enforce role checks. Only architects can approve design. Only DevOps can approve prod deploys. JWT-verified.
📺
Event Types — What the Dashboard Receives
# Every event is typed and streamed in real-time

on_chain_start → Agent {name} started — input: {story_id}
on_tool_start → Tool {tool_name} called — args: {args}
on_tool_end → Tool {tool_name} returned — result: {result}
on_llm_start → LLM {provider}/{model} invoked — tokens: {n}
on_llm_end → LLM returned — latency: {ms}, cost: ${cost}
on_chain_end → Agent {name} completed — output: {summary}
on_chain_error → Agent {name} FAILED — error: {trace}
on_interrupt → Human gate triggered — waiting for: {role}
on_resume → Human approved — resuming from: {node}
